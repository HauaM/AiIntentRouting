1 · 검토 배경 및 평가 기준

Intent Routing Service는 사용자 질문에 직접 답하지 않고 intent · route_key · decision(confident/clarify/fallback/off_topic/risk)을 판정하는 FastAPI 백엔드입니다. 관리 콘솔은 4개 역할(시스템관리자·서비스개발자·운영자·감사/보안)이 카탈로그 관리, 테스트/배포, 운영 모니터링, 감사를 수행하는 내부 백오피스입니다. 따라서 템플릿은 화려함이 아니라 업무 흐름의 정확한 표현·신뢰감·폐쇄망 적합성으로 평가합니다.

차분·신뢰감 있는 톤 CRUD·테이블·필터·상세·승인 구현 용이성 감사·배포 워크플로우 표현력 대시보드/메트릭 확장성 역할별 메뉴/화면 제어 FastAPI REST 연동 적합성 폐쇄망/오프라인 빌드

3 · 최종 추천 — Ant Design Pro v6

Pro v6는 Ant Design 5 위에 ProLayout · ProTable · ProForm · ProDescriptions · ProCard 등 백오피스 전용 고수준 컴포넌트(ProComponents)를 얹은 공식 엔터프라이즈 템플릿입니다. 본 콘솔이 필요로 하는 화면 유형 — 목록(필터/정렬/페이지네이션) → 상세 → 편집 → 승인 → 배포 — 이 모두 표준 부품으로 매핑됩니다.

  • 업무 화면 적합도 — ProTable 하나로 Intent Catalog/Runtime Logs/Audit Logs의 검색·필터·컬럼 토글·서버 페이지네이션을 선언적으로 구성. request 함수만 FastAPI에 연결하면 됩니다.
  • 워크플로우 표현 — Steps · Result · Descriptions · Drawer · Modal.confirm 조합으로 Examples 승인, Release 활성화 같은 다단계/확인 플로우를 정석적으로 표현.
  • 권한 제어 — Pro의 access + 라우트 authority 모델로 역할별 메뉴/버튼/컬럼 노출을 깔끔하게 통제.
  • FastAPI 연동 — OpenAPI 스키마로 TS 타입/클라이언트 생성 → ProTable/ProForm에 그대로 주입. 토큰/세션은 요청 인터셉터로 일괄 처리.
  • 폐쇄망 적합 — Umi/Vite 정적 빌드 산출물을 사내 레지스트리로 오프라인 설치·배포 가능. 외부 폰트/CDN만 사내 자원으로 치환하면 됨.
  • 톤 — 기본 테마가 절제된 엔터프라이즈 룩이라 마케팅 사이트 느낌이 없음. 금융권 운영/감사 콘솔에 자연스럽게 맞음.
유의 — 차트는 별도(@ant-design/charts/ECharts)로 추가해야 하며, 폐쇄망에서는 해당 패키지/폰트의 오프라인 번들링을 사전 검증해야 합니다.

4 · 대안 2종

Refine + Ant Design
데이터 중심 내부도구 프레임워크
장점
  • useTable/useForm/useList 등 데이터 훅으로 CRUD 보일러플레이트 최소화
  • 인증·권한(accessControl)·감사(audit-log) provider가 1급 개념
  • REST dataProvider로 FastAPI 연결이 빠름
단점
  • Refine 추상화 학습비용 + 프레임워크 종속
  • 비정형 화면(Release diff 등)은 결국 직접 구현
  • 의존성 트리가 커 폐쇄망 빌드 검증 부담 ↑
AntD 기본 + 자체 스캐폴딩
antd 컴포넌트만 + 자체 레이아웃
장점
  • 의존성 최소·완전 통제 → 보안 검수/감사 용이
  • 화면 구조를 업무에 100% 맞춤 설계
  • 오프라인 빌드 가장 단순
단점
  • 레이아웃·테이블 툴바·폼 패턴을 직접 구축(초기 비용 큼)
  • 팀 간 화면 일관성 유지에 규율 필요
  • Pro가 공짜로 주는 패턴을 다시 만들게 됨
Ant Design X (참고) — 대화형 AI UI(Bubble, Sender, Conversations)에 특화. 본 콘솔의 운영/감사 성격과 결이 다르므로 베이스 부적합. 단, Dify Integration 화면에서 HTTP Request node 설정 예시나 프롬프트/응답 샘플을 보여주는 보조 위젯 정도로만 선택 적용을 권합니다.

7 · 색상 / 톤앤매너

9 · 피해야 할 UI 패턴

  • 챗봇/마케팅 무드 — 큰 히어로, 그라데이션 배경, 일러스트, 말풍선 중심 레이아웃. 운영 콘솔엔 부적합.
  • raw query 무방비 노출 — 기본은 항상 마스킹. 원문은 2인 승인 + 감사 기록 후에만, 시한부 열람.
  • 되돌릴 수 없는 작업의 1-클릭 실행 — release 활성화·key revoke·승인은 Modal.confirm/Popconfirm + 영향 요약 필수.
  • 과도한 색·아이콘 남발 — 액센트는 핵심 액션에만. decision/status 외 장식성 색·이모지·불필요한 통계 타일 금지.
  • 무한 스크롤 테이블 — 감사/로그는 페이지네이션 + 명시적 기간 필터 + 내보내기로 재현성 확보.
  • 권한 무시 전체 메뉴 노출 — 역할별로 메뉴/버튼/컬럼을 숨김. "보이지만 비활성"보다 "안 보임"을 기본으로.

10 · 최종 추천 & 시작 순서

Ant Design Pro v6 로 시작하세요. ProComponents가 본 콘솔의 목록→상세→편집→승인→배포 패턴을 표준 부품으로 커버하고, 차분한 엔터프라이즈 톤과 폐쇄망 정적 빌드에 모두 맞습니다. ConfigProvider로 네이비 토큰만 입히면 톤앤매너가 즉시 정렬됩니다.

권장 구축 순서 (얇게 끝까지 한 번 → 확장)

1
셸 + 인증/권한 + Services·API Keys — ProLayout, access 모델, ConfigProvider 토큰. 시스템관리자 흐름으로 골격 확정.
2
Intent Catalog (목록 + 상세/편집) — ProTable/ProForm의 CRUD 레퍼런스 화면. 이후 모든 화면의 패턴 기준이 됨.
3
Runtime Logs + Dashboard — 마스킹/필터/Drawer 확립 후 같은 데이터로 대시보드 메트릭 구성.
4
Examples 승인 → Releases → Audit — Steps/Modal.confirm/2인 승인 등 워크플로우·감사 흐름을 마지막에 통합.

CSV Test Runs·Policy Versions·Security/Key Summary·Dify Integration은 2~4단계의 패턴을 재사용해 확장합니다. Dashboard와 Intent Catalog를 가장 먼저 시각적으로 확정하면 이해관계자 합의와 이후 화면 일관성 확보에 가장 효과적입니다.
